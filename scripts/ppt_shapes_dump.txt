0|Introduction 
1|3
2|Career platforms often give generic advice without clear job-fit reasoning.
PathFinder AI is a full-stack web platform (React + FastAPI + SQLite) for students and job seekers.
Two roles: job seeker and recruiter.
The system: KNN career suggestions; job ranking (Doc2Vec + importance-weighted similarity); Gemini roadmaps with gap analysis; contextual bandit adapts saved roadmaps from task feedback.
Stack: React, FastAPI, SQLite, ChromaDB (optional RAG), XGBoost, gensim Doc2Vec, Google Gemini.
--- 4
0|Problem statement:
• Keyword-only matching lacks explainability and structured learning paths.
• User skills are not mapped clearly to real JD requirements.
• Roadmaps are static and do not adapt when tasks are too hard or irrelevant.
• Recruiters and seekers need one platform for jobs, matching, and guidance.

Objectives:
• Build JWT-authenticated user and recruiter workflows.
• Resume upload with Gemini + lexicon-based skill extraction.
• KNN career recommendations; Doc2Vec + Model 2 weighted job ranking.
• Model 1: classify JD skills as core / important / supporting / optional (saved per job).
• Gemini JSON roadmaps; Phase 2 contextual bandit (7 actions) on task feedback.

--- 5
0|Pipeline (as implemented):
• Skills: datasets/skills.json lexicon + competency extractor; resume via Gemini (local fallback on quota).
• Model 1 (XGBoost): JD skill importance — stored once per job until JD changes.
• Model 2: similarity score (0–100) from weighted matched/missing skills + experience (no high/medium/low labels).
• Doc2Vec baseline: semantic match + 70/30 skill overlap (fallback matcher).
• Gemini: structured roadmaps (role summary, gap analysis, phased tasks).
• Contextual bandit (ε=0.15, 10-D state, 7 roadmap actions) — adapts learning plans, not job lists.
1|Proposed Solution Overview 
2|5
--- 6
0|Key ideas:
• Three job-matching layers: Doc2Vec (semantic), Model 1 (JD parsing), Model 2 (weighted user–job similarity).
• Gemini + optional ChromaDB RAG for chat and context-aware guidance.
• Contextual bandit (not full MDP RL) edits roadmaps after Complete / Too Hard / Too Easy / Skip & Regenerate.

Workflow:
1. Register → profile + optional resume.
2. KNN career recommendations; search/browse jobs; AI job matching.
3. Generate roadmap for a job → saved per job (persists on refresh).
4. Task feedback → log → recommend → adapt (Phase 2 APIs).
5. Recruiter posts jobs with JD, skills, salary, remote options.

--- 12
0|Datasets & setup:
• 799 job postings; 12,375 (job, skill) rows; 460-skill lexicon

Model 1 — XGBoost skill importance (test n=1,857):
• Accuracy 98.60% | macro-F1 97.32% (P 97.82%, R 96.83%)
• Per-class F1: core 0.97 | important 0.95 | supporting 0.99 | optional 0.99
• Top features: context_score 41%, section_score 37%

Model 2 — similarity matching (user vs job, not fit classes):
• similarity_score 0-100 from weighted matched vs missing skills (Model 1 tiers)
• Weights: core 3.0 | important 2.0 | supporting 1.0 | optional 0.5
• Blend: 72% skills | 20% experience | 8% profile-JD lexical; core-miss penalty
• Outputs: ranked jobs, matched/missing/critical skills — no high/medium/low labels

Also: Doc2Vec semantic baseline; roadmap bandit (epsilon=0.15, 7 actions, 10-D state)

0|Live demo flow (React :3000 + FastAPI :8001):
1. Register -> upload resume -> profile skills (Gemini + lexicon fallback)
2. Career recommendations (KNN on resume embedding)
3. Search Jobs -> filter SQLite board; open job detail
4. Job Matching -> ranked list with similarity %, matched & missing skills
   (Model 2 primary; Doc2Vec fallback if profile sparse)
5. Generate roadmap -> JSON phases saved per job (refresh-safe upsert)
6. Roadmap feedback: Complete / Too Hard / Too Easy / Skip
   -> log -> recommend -> adapt (bandit action + Gemini task rewrite)
7. Compare up to 3 jobs side-by-side; recruiter posts JD -> Model 1 skills cached

Explainability shown to user: skill tiers, similarity %, gap analysis, adaptation reason
